## Техническая спецификация информационной системы "Портал Развития для ТСЧ"

### 1. Общие сведения

#### 1.1. Наименование системы
**Портал Развития для ТСЧ** (Целевая информационная система для автоматизации процессов развития торговой сети "Чижик") [БРИФ_ Портал Развития для ТСЧ-v29-20251204_170839.docx]

#### 1.2. Назначение и цели создания системы
**Назначение:** 
Система предназначена для автоматизации процессов подготовки и согласования документов для защиты проектов развития ТСЧ на Инвестиционном комитете (ИК), включая централизованное ведение, мониторинг и контроль процессов подготовки и согласования документов.

**Цели:**
1. Обеспечить контролируемость проектов открытия новых магазинов ТСЧ в прогнозируемых объемах открытий
2. Повысить эффективность проектов открытия новых магазинов ТСЧ
3. Сократить сроки открытия магазинов ТСЧ [БРИФ_ Портал Развития для ТСЧ-v29-20251204_170839.docx]

### 2. Характеристики объекта автоматизации

#### 2.1. Краткое описание бизнес-процессов
Система автоматизирует следующие ключевые процессы:
1. **Поисковая заявка** – создание карточки проекта, синхронизация с ГИС, определение участников
2. **Аудит** – планирование, проведение, фиксация результатов
3. **Материалы к ИК** – подготовка визуализаций, планировок, расчетов бюджетов, согласование документации
4. **Отправка данных** – формирование инвестиционной модели и презентации, синхронизация статусов

В текущей ситуации (As Is) процессы ведутся вручную: согласование документов в почте, ручное формирование презентаций (40 слайдов), отсутствие централизованного мониторинга. [БРИФ_ Портал Развития для ТСЧ-v29-20251204_170839.docx]

### 4. Нефункциональные требования

#### 4.1. Требования к производительности
- Время генерации отчетов: не более 10 секунд при объеме данных до 1000 проектов
- Время ответа на запросы: до 2 секунд (95% запросов)
- Максимальное количество одновременных сессий: 450 пользователей (максимальное количество участников)

*Экспертное предположение: Требования уточняются на этапе разработки архитектуры.*

#### 4.2. Требования к надежности и доступности
- Время предоставления сервиса: **24x7** (круглосуточно)
- График работы поддержки: **ПН-ЧТ 09:00-18:15, ПТ 09:00-17:00** (по московскому времени)
- Время решения обращений: **4-8-16-24** (Критичный 4, Важный 8, Средний 16, Низкий 24)
- Целевой SLA: **95%** (доля выполненных в срок обращений) [БРИФ_ Портал Развития для ТСЧ-v29-20251204_170839.docx]

#### 4.3. Требования к безопасности
- **Безопасность данных:**  
  Не влияет на процессы хранения, обработки или передачи персональных данных (ПДн)  
  Не влияет на процессы, связанные с шифрованием

#### 4.4. Требования к масштабируемости

- Поддержка 1000+ проектов в системе

#### 4.5. Требования к интеграционным потокам

##### 4.5.1. Интеграция с ГИС (Геоинформационная система)
**Направление потока:** Двунаправленная интеграция (ГИС → Портал и Портал → ГИС)

**1. Синхронизация данных проекта:**
- **Автоматическая передача данных при переходе статусов:**
  - При переходе проекта в ГИС в статус "Аудит" → Портал автоматически получает данные о проекте, ответственных специалистах и макрорегионах [БРИФ_ Портал Развития для ТСЧ-v29-20251204_170839.docx]
  - При фиксации фактической даты аудита → Портал отправляет статус "Аудит завершен" в ГИС
  - При переходе в статус "Бюджет сформирован" → Портал отправляет данные для инвестиционной модели и презентации в ГИС [БРИФ_ Портал Развития для ТСЧ-v29-20251204_170839.docx]

- **Синхронизация ключевых статусов:**
  - При переходе проекта в ГИС в статусы: "Утвержден ИК ТС", "Утвержден РР", "Подписан договор", "РСР", "Открыт", "Слетел" → автоматическая передача статуса на Портал
  - На Портале фиксируется дата-время изменения статуса в карточке проекта [БРИФ_ Портал Развития для ТСЧ-v29-20251204_170839.docx]

**2. Требования к API:**
- Использование REST API для обмена данными
- Поддержка транзакций: данные о проекте и статусы должны обрабатываться атомарно
- Механизм обработки ошибок: ретрансляция запросов, логирование, уведомления администраторов
- Поддержка аутентификации через корпоративный SAML

**3. Частота синхронизации:**
- Реактивная синхронизация (при изменении статуса)
- Периодическая синхронизация данных (1 раз в 15 минут) для актуализации макрорегионов и привязки специалистов

**4. Документация:**
- Документация API должна быть размещена в Kaiten
- Поддержка OpenAPI (Swagger) для тестирования

**5. Структура полей данных:**
- **Данные, передаваемые из ГИС в Портал:**
  - *Код проекта* (ID)
  - *Статус проекта* (текст)
  - *Код ГИС* (текст)
  - *Информация об объекте* (адрес, площадь, регион)
  - *Список ответственных* (роли, ФИО)
  - *Дата перехода в статус* (дата-время)
  - *Комментарий* (текст)
  - *Документы* (ссылки на документы в ГИС)
  - *Данные о макрорегионах* (структура, привязка специалистов)

- **Данные, передаваемые из Портала в ГИС:**
  - *Код проекта* (ID)
  - *Статус проекта* (текст)
  - *Дата перехода в статус* (дата-время)
  - *Данные для инвестиционной модели* (бюджеты, РТО)

  - *Согласованные данные* (список согласований)
  - *История изменений* (дата, пользователь, описание)
